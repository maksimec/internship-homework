## 1. Автоматизація та Systemd

### Disk та RAM моніторинг (Timer approach)
Для моніторингу диска та оперативної пам'яті було обрано підхід із використанням **systemd timers**. 
- Скрипти `disk_monitor.sh` та `ram_monitor.sh` виконують разову перевірку (`Type=oneshot` у `.service` файлах).
- Відповідні таймери (`disk-monitor.timer` та `ram-monitor.timer`) налаштовані на запуск цих сервісів щогодини.

### Log Watcher (Long-running process approach)
Для моніторингу логів у реальному часі використано підхід **long-running service**.
- Скрипт `log_watcher.sh` використовує `tail -F` для безперервного читання логів `disk_monitor.log` та `ram_monitor.log`.
- Файл `log-watcher.service` запускає цей скрипт як постійний фоновий процес (без використання таймера).

## 2. Advanced: Налаштування поштового сервера та DNS

В рамках додаткового (Advanced) завдання було налаштовано повноцінну email-інфраструктуру для відправки повідомлень із сервера.

### Виконані кроки на сервері:
1. **Встановлення пакетів:**
   \`\`\`bash
   apt update
   apt install postfix mailutils opendkim opendkim-tools
   \`\`\`
   *(Під час встановлення Postfix обрано конфігурацію "Internet Site").*

2. **Генерація та налаштування DKIM-ключів:**
   \`\`\`bash
   mkdir -p /etc/opendkim/keys/server.maksimecv.pp.ua
   opendkim-genkey -d server.maksimecv.pp.ua -D /etc/opendkim/keys/server.maksimecv.pp.ua -s mail
   chown -R opendkim:opendkim /etc/opendkim/keys/
   \`\`\`

3. **Конфігурація OpenDKIM та Postfix:**
   - OpenDKIM налаштовано на прослуховування порту `inet:127.0.0.1:8891`.
   - У файл `/etc/opendkim.conf` додано згенерований ключ для домену `server.maksimecv.pp.ua`.
   - У `main.cf` Postfix додано інтеграцію з OpenDKIM через параметри `smtpd_milters` та `non_smtpd_milters`.

4. **Налаштування DNS-записів:**
   Для забезпечення доставки листів (уникнення потрапляння в спам) для субдомену `server.maksimecv.pp.ua` були налаштовані наступні записи:
   - **SPF:** TXT-запис, що дозволяє IP-адресі сервера відправляти пошту від імені домену.
   - **DKIM:** TXT-запис (із файлу `mail.txt`), що містить публічний ключ для верифікації криптографічного підпису листів.
   - **DMARC:** TXT-запис, що визначає політику обробки листів, які не пройшли перевірку SPF або DKIM.
   - **PTR (Reverse DNS):** На рівні хостинг-провайдера налаштовано зворотний запис, який вказує з IP-адреси на `server.maksimecv.pp.ua`.

### Тестування:
Для перевірки у скриптах моніторингу пороги спрацювання були тимчасово знижені до **1%**, щоб згенерувати тестові листи. 
- Успішне отримання листів підтверджено (див. `screenshots/received_letters.png`).
- Валідність DKIM, SPF та DMARC підтверджено заголовками отриманих листів (див. `screenshots/letter_header.png`).
- Додатково конфігурацію перевірено через сервіс **mail-tester.com** (оцінка висока, знято бали лише за відсутність MX-запису, оскільки налаштовувалася виключно відправка). Посилання на звіт: [Mail-Tester Report](https://www.mail-tester.com/test-c5hmbo7bc).
- За необхідності можу надати доступ по SSH до сервера для перевірки працездатності, а також налаштувань (для цього чекаю SSH pubkey в ПП).

## 3. Відповіді на теоретичні питання (Research)

### Inodes Research (див. `screenshots/inodes.png`)
**1. Що станеться, якщо inode закінчаться, але дисковий простір ще є?**
Не буде змоги створювати нові файли (директорії, сімлінки, сокети йдуть в цю ж категорії, оскільки в Linux це все є файлами). При спробі створити файл буде фіксуватись помилка `No space left on device`, навіть якщо `df -h` буде показувати, що вільний дисковий простір ще є.

**2. Чому при створенні великої кількості порожніх файлів змінюється кількість inode, але майже не змінюється зайнятий диск?**
Кожен файл, незалежно від його розміру, потребує рівно одного inode для зберігання метаданих (права доступу, власник, час створення). Порожній файл не використовує блоки даних на диску для зберігання вмісту (тому об'єм вільного дискового простору по `df -h` не змінюється), але він "споживає" один inode зі структур файлової системи.

**3. Як знайти директорію з найбільшою кількістю файлів?**
Якщо багато вільного часу та не має бажання скористатись зручнішим варіантом, можна від / йти, використовуючи:
\`\`\`bash
du --inodes -s /*
\`\`\`
Або ж можна скористатися комбінацією команд `find`, `cut`, `sort` та `uniq`. Наприклад:
\`\`\`bash
find /path/to/search -xdev -type f | cut -d "/" -f 1-2 | sort | uniq -c | sort -n
\`\`\`
Або швидший варіант з використанням `find` та `wc` у циклі:
\`\`\`bash
find / -xdev -type d -exec sh -c 'echo "$(find "$0" -maxdepth 1 | wc -l) $0"' {} \; | sort -rn | head -n 10
\`\`\`
Загалом подібних рішень в інтернеті та в ШІшках дуже багато.

### Sticky Bit (див. `screenshots/sticky-bit.png`)
**1. Що означає `t` у правах доступу `drwxrwxrwt`?**
Символ `t` позначає встановлений **Sticky Bit** (разом із правом виконання для всіх інших користувачів — `x`). Якщо б права виконання не було, відображалася б велика літера `T`.

**2. Де sticky bit використовується за замовчуванням?**
Як усталене налаштування sticky bit встановлений на системні директорії тимчасових файлів, такі як `/tmp` та `/var/tmp`.

**3. Що станеться, якщо інший користувач спробує видалити твій файл у такій директорії?**
Операція буде відхилена системою із помилкою `Operation not permitted`. Зі встановленим sticky bit на директорії, видалити або перейменувати файл може лише власник цього файлу, власник самої директорії або суперкористувач (`root`), навіть якщо права доступу до директорії (`777`) дозволяють запис усім.

**4. Навіщо sticky bit потрібен у спільних директоріях?**
Він забезпечує безпеку та ізоляцію в директоріях, до яких мають доступ на запис багато користувачів (як `/tmp`). Sticky bit запобігає ситуаціям, коли один користувач (навмисно чи випадково) видаляє або змінює файли, що належать іншим користувачам.
